Every binary protocol eventually needs the same things: a way to mark where a packet starts and ends, a checksum to catch corruption, a header encoding that is byte-order safe, a way to describe the payload layout, and an incremental reader that handles partial data from a socket. These are not interesting engineering problems. Every network engineer has solved them, and solved them again, and then inherited someone else's solution and solved them a third time.

The usual answer is a serialisation framework. Protobuf. MessagePack. Cap'n Proto. They solve all five problems, and more: reflection-based field discovery, versioned schema registries, code generation from `.proto` files. That is fine when the protocol is yours to define. It is not fine when you are implementing an existing wire format, need full control over the byte layout, or are targeting a microcontroller with 4 KB of RAM.

What is left is `encoding/binary` and manual framing. It is correct. It is also a wall of `binary.Read` calls, hand-rolled CRC loops, and a partial-read state machine that everyone rebuilds from scratch every project.

`wireframe` is a toolkit that covers that middle ground.

---

## The usual answers fall short

`encoding/binary` handles byte-order encoding but knows nothing about framing. It has no concept of a packet boundary and does not buffer partial reads. Every team ends up writing the same wrapper: a `readFull` loop, a length-prefix field, a CRC check. That wrapper is never extracted into anything reusable.

Serialisation frameworks like Protobuf handle all of it, but they own the wire format. The schema is authoritative; the generated code decides the byte layout, not you. Implementing an existing protocol (Modbus RTU, a legacy binary format, an embedded device protocol) is not possible when the framework owns the layout. You also pick up a code generation step, a third-party schema compiler, and a generated file that drifts every time the `.proto` changes.

Rolling it all by hand each time is viable, but the state machine for incremental parsing is subtle. Getting `io.ErrUnexpectedEOF` right, distinguishing a clean EOF at a frame boundary from a half-written frame, avoiding allocations in the hot read loop: these are not hard problems, but they require attention every time, and the cost is paid silently when someone gets it wrong.

## What `wireframe` does

A `Format` describes the structure of one frame type. You supply the prefix bytes, the ordered list of header fields, the checksum algorithm, and the maximum payload size. The codec does the rest.

```go
format := &frame.Format{
    Prefix: []byte{0xAA, 0x55},
    Header: []frame.Field{
        frame.Uint8("version"),
        frame.Uint8("type"),
        frame.Uint8("flags"),
        frame.PayloadLength("length"),
    },
    Checksum:   checksum.CRC32IEEE(),
    MaxPayload: 64 * 1024,
}
```

Encoding populates a `Values` and calls `Encode`. `Values` is a value type; declare it on the stack, call `Reset` between frames, and no heap allocation occurs.

```go
var vals frame.Values
frame.Set(&vals, "version", uint8(1))
frame.Set(&vals, "type",    uint8(0x42))
frame.Set(&vals, "flags",   uint8(0))

encoded, err := format.Encode(nil, payload, &vals)
if err != nil {
    return err
}
```

`Encode(dst []byte, ...)` follows the append pattern. Pass `nil` to allocate; pass a pre-grown slice to reuse it.

`stream.NewReader` wraps any `io.Reader`, buffers data, handles partial reads, and returns complete frames one at a time. For the lifetime of a connection it performs zero per-frame heap allocations.

```go
sr := stream.NewReader(conn, format)

for {
    payload, vals, err := sr.Next()
    if errors.Is(err, io.EOF) {
        return nil
    }
    if err != nil {
        return err
    }
    switch vals.Uint8("type") {
    case 0x42:
        handle(payload, vals)
    }
}
```

`vals` is valid until the next call to `Next`. Copy it if the header needs to outlive one iteration.

The `Values` API is correct and allocation-free, but field names are strings checked only at runtime. `TypedFormat[T]` binds the codec directly to a struct, using pointer-to-field accessor functions as the bridge.

```go
type Header struct {
    Version uint8
    Type    uint8
    Flags   uint8
    Length  uint16
}

var msgFormat = frame.NewTypedFormat[Header](
    frame.WithPrefix([]byte{0xAA, 0x55}),
    frame.Uint8Field("version", func(h *Header) *uint8  { return &h.Version }),
    frame.Uint8Field("type",    func(h *Header) *uint8  { return &h.Type }),
    frame.Uint8Field("flags",   func(h *Header) *uint8  { return &h.Flags }),
    frame.PayloadLengthField("length", func(h *Header) *uint16 { return &h.Length }),
    frame.WithChecksum(checksum.CRC32IEEE()),
    frame.WithMaxPayload(64*1024),
)

// Encode
hdr := Header{Version: 1, Type: 0x42}
encoded, err := msgFormat.Encode(nil, payload, &hdr)

// Decode
var hdr Header
payload, n, err := msgFormat.Decode(src, &hdr)
if errors.Is(err, frame.ErrChecksum) {
    return fmt.Errorf("integrity check failed: %w", err)
}
```

`TypedFormat[T]` wraps the untyped `Format`. Internally it translates between `*T` and `*Values` using the accessor functions. The untyped API remains the foundation; this is a compile-time convenience layer on top of it.

`schema` describes the payload layout. Independent from `frame`; use one, both, or neither.

```go
s := schema.New(
    schema.Uint8("command"),
    schema.Uint16BE("seq"),
    schema.VarBytes("data", "seq"),
)

var vals schema.Values
schema.Set(&vals, "command", uint8(0x01))
schema.Set(&vals, "data", []byte("hello"))

encoded, err := s.Encode(nil, &vals)
```

## The Values design

The most direct alternative to `Values` is `map[string]any`. Every field value goes through an interface, which means a heap allocation per field per frame. At 100K frames per second on a busy bus that is a meaningful GC load.

`Values` uses a fixed-size inline array: 16 slots of `(string, uint64|[]byte)`. Integer values are stored as `uint64` with no interface boxing. String lookup is a linear scan over `keys[0:n]`. Protocol headers are almost never more than eight fields; a linear scan over eight strings is faster than a map lookup at that scale.

Callers declare `Values` on the stack. `Reset` zeroes the counter. The struct never escapes to the heap in the common case. `stream.Reader` holds a `Values` as a struct field, so the reader's allocation budget for a running connection is exactly one: the initial `NewReader` call.

The constraint is the point: `Values` is not a general key-value store. It is sized for the thing it is actually used for.

## What was not built

Struct tags and `reflect` would let callers write `Frame(format, &myStruct)` without accessor functions. That path hides the encode/decode logic from the toolchain, breaks `go tool` analysis, and costs an `import "reflect"` for a problem that accessor functions solve at zero runtime cost. Accessor functions are more code to write and more code to read. That is the trade-off; it is the right one.

The format is declared with explicit byte order at every field (`Uint16BE`, `Uint32LE`). A detection heuristic would silently pick the wrong order on a protocol that mixes endianness. You declare it; the codec follows.

Encryption and compression belong above `wireframe`, applied to the payload before it reaches the codec. Mixing them into the framing layer would force every caller to carry the cost, whether they need it or not.

`Encode` follows the append pattern; callers write the result to the connection themselves. A `Writer` wrapping `io.Writer` was considered and not built because it would own the write call and remove the caller's ability to batch frames or control flush timing.

Version fields and type dispatching are values in the header. What to do with them is your logic, not the codec's.

The full implementation is 13 files across 7 packages. Zero external dependencies. Everything `wireframe` needs is already in `io`, `bufio`, `bytes`, `encoding/binary`, `hash/crc32`, `hash/crc64`, and `hash/adler32`. Importing an external library for any of those would mean subscribing permanently to someone else's release cycle for problems the standard library already solves. Read the source, vendor it, take it. The code is small enough to hold in one sitting.
